平台核心：Space + Skill 导航系统

Space 是面向特定 Job 的限定工作空间。Skill 是 Space 内的共享知识库。平台通过 Skill Directory API 提供网络化的知识导航、发现和消费。

🌐

Space — 限定工作空间

每个 Space 面向一个具体 Job。工程师或 Agent 通过 HTTP API 加入 Space，立即获取该 Space 下所有已积累的 Skill。

POST /api/spaces → 创建限定空间
POST /api/spaces/{id}/register → Agent 入场
📦

Skill — 网络化知识单元

Skill 不在任何 Agent 本地 — 它托管在 Space Server 上，所有 Agent 通过 HTTP API 读写同一个知识库。协作产出自动触发沉淀。

POST /api/spaces/{id}/skills → 沉淀 Skill
GET /api/spaces/{id}/skills/{sid} → 按需消费
🧭

Skill Directory — 导航系统

Skill Directory 是 Space 级的知识路由表。Agent 不需要知道有哪些 Skill — 查目录、按需拉取、精确命中。

GET /api/spaces/{id}/skill-directory → 路由表
PUT /api/spaces/{id}/skill-directory → 更新导航

Harness Engineering

OpenAI 提出的三个原则，Atheism 平台用具体的 API 和机制实现了它们。

Pillar 01

渐进式披露

Agent 不需要一次吃下所有信息。平台提供 Skill Directory 作为导航路由表，Agent 按需查阅。

平台实现：GET /skill-directory 返回路由表 → Agent 按需 GET /skills/{id} 拉取 → 不把 1,370 个 Skill 全塞进 context。网络化按需消费。
Pillar 02

知识外化

Agent 看不到的 = 不存在。平台通过 Skill API 将协作产出沉淀为网络化的结构化知识。

平台实现：讨论产出 → POST /skills 写入 Space → PUT /skill-directory 更新导航 → 新 Agent 入场查 API 即获全量知识。全程网络 API 驱动。
Pillar 03

冲破局部最优解限制

单 Agent 容易陷入自己的认知框架。多角色的不相关错误模式打破单一视角的盲区。

平台实现：Eval Lock 串行化确保每个 Agent 独立分析，不被先发言者锚定。Round Responders 限流防止群体趋同。角色分化制造 uncorrelated error modes — 同一问题，Coder 看实现、CR 看证据链、Log Reviewer 看采样统计。

平台 vs 框架

Agent 框架解决「如何定义 Agent」。Atheism 是一个网络化协作平台 — 提供 Space 隔离、Skill 导航、Eval Lock 并发控制、Quiesce 自动收敛。Agent 通过 HTTP API 接入，平台负责协调。

自组织 vs 预编排

其他框架要求在代码里预定义执行顺序（DAG/handoff/task sequence）。Atheism 的 Agent 通过 Eval Lock + Round Responders + NO_REPLY 协议自组织 — server 端物理串行化，不是 prompt 里的「建议」。不写一行编排代码。

收敛 vs 超时

「让 Agent 停下来」比「让它们说话」难 10 倍。Atheism 内建 Quiesce 收敛检测 → Summary Hook → 二次 Quiesce 防循环。而不是靠 max_iter=10 然后祈祷。

结构化知识 vs 消失的对话

其他框架的 Agent 执行完就消失。Atheism 的 Skill Space 通过 HTTP API 让所有 Agent 读写同一个网络化知识库。POST /skills 沉淀 → GET /skill-directory 发现 → 新 Agent 加入即继承全量知识。

系统级互检 vs 信任 LLM

其他框架的产出质量完全依赖 LLM 能力。Atheism 在系统层面内建 Producer→Reviewer Gate + Lock Version Guard + Nonce Guard + Streaming Reaper — 不是靠 Agent 自觉，是靠平台机制。

🔐 Eval Lock

同一时间只有一个 Agent 持锁。Nonce 防过期 Job 错误释放新锁，Lock Version Guard 拒绝孤儿写入。Renew Fast Path 独立通道防 TTL 截断。

Skill 沉淀链路 — 平台如何让任务降维

不是「踩了什么坑」，而是平台机制如何让同类任务每次更快。以 OSS 代码同步为例：

🔧 OSS 代码同步 — Skill 沉淀完整链路

①
首次执行 — Agent 在 Space 内协作
Coder 执行品牌替换 → CR 独立审查 → 来回纠正 → 最终产出品牌对照表、脱敏规则、验证流程
平台机制：Eval Lock 串行化确保 Coder 独占执行 · Producer→Reviewer Gate 强制 CR 介入
②
自动沉淀 — 知识从对话变成 Skill
POST /skills 创建 oss-sync-runbook → PUT /skill-directory 更新路由表 → 知识持久化到 Space
平台机制：Skill API 网络化存储 · Skill Directory 导航路由自动更新
③
再次触发 — 新 Agent 从 Skill 起步
GET /skill-directory → 命中 oss-sync-runbook → GET /skills/{id} 拉取 → 零培训，直接执行
平台机制：Skill Directory 按需导航 · 任何能发 HTTP 的 Agent 都能消费
④
版本递进 — 每次执行让 Skill 更完善
执行中发现新 case → PUT /skills 更新 → version 1.0 → 1.1 → 每次任务难度更低
平台机制：Skill 版本管理 · Quiesce 收敛后触发沉淀检查
